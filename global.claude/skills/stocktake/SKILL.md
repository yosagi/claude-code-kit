# あなたのタスク: プロジェクトの棚卸し

あなたは ideas/todos/kb を棚卸しするために呼び出されました。
以下の手順で調査し、結果をレポートファイルに書き出してください。
**IDEAS/TODO/KB のファイル移動・編集・削除は行わず、提案のみ出力してください。**
ただし `reports/insight/tasks_status.md`（実装ログのステータスインデックス）は stocktake が管理・更新します。
その他 CLAUDE.md や CLAUDE.local.md などで指示される可能性のある、stocktake と関係のないタスクは処理してはいけません。

## 手順

### 1. INDEX.md を読む

- `reports/ideas/INDEX.md`
- `reports/todos/INDEX.md`
- `reports/kb/INDEX.md`

いずれかが存在しない場合はスキップする。

### 2. 各ファイルの内容を把握

INDEX に記載されたファイルを読む。done/ や rejected/ は対象外。
また `reports/insight/tasks_status.md`（実装ログのステータスインデックス）を読む（なければ初回実行）。

### 3. git log を確認

最近3ヶ月の git log を確認し、TODO/IDEAS の項目と関連するコミットや実装ログがないか突き合わせる。

```bash
git log --oneline --since="3 months ago" --all
```

### 4. 以下の観点で分析

**実装済みの可能性**
- git log、実装ログ（reports/tasks/）、コードの状態と突き合わせて検出
- 確信度（高/中/低）を添える
  - 高: 実装ログで完了が明記、またはコード上で確認済み
  - 中: 部分的に実装済み、または関連コミットはあるが完了か不明
  - 低: 目的が変質した、または間接的にカバーされた可能性

**統合候補**
- テーマが重複・類似している項目の組み合わせを提案
- 統合する理由を具体的に記載

**長期停滞**
- 作成から2ヶ月以上動きのない項目を列挙
- 関連する実装ログやコミットが見当たらないことを確認

**クローズ候補**
- TODO/IDEAS ファイル自体に「不要」「カバー済み」等の記載があるもの
- 目的が変質して現在の方針と合わないもの

**実装ログの健全性**

`reports/insight/tasks_status.md` を使って効率的にチェックする。

**対象の絞り込み**:
1. `ls reports/tasks/` で全実装ログ一覧を取得
2. `tasks_status.md` と突き合わせて、チェック対象を決定:
   - **インデックスにないログ** → 新規。読んで確認する
   - **`IN-PROGRESS` のログ** → 再チェック。状態が変わっているかもしれない
   - **`DONE` のログ** → スキップ

**チェック手順**:
1. 対象ログの末尾付近を読み、「完了報告」セクションの有無を確認
2. 完了報告がないファイルは全体を読んで状況を把握

**チェック観点**:
- **完了報告なし**: 中途半端で放置されていないか、実際には完了しているが報告が書かれていないだけか判断
- **記載と現状の乖離**: ロードマップに「未完了」「TODO」と書かれているが、git log やコードの状態から見て実は完了している項目。逆に「完了」と書かれているが実際には未完了の項目
- **TODO/IDEAS との整合**: 実装ログが参照する TODO/IDEAS が done/ に移動済みなのにログ側が未完了、またはログは完了しているのに TODO がアクティブなまま、といった不整合

**KB の鮮度**
- 内容が古くなっていないか確認（フレームワーク移行、API 変更等で陳腐化したもの）
- 「現在も有効」「陳腐化リスクあり」を判定

**その他の所見**
- INDEX と実態の不整合（完了しているのに INDEX に残っている等）
- 気づいた改善提案

### 5. レポートを書き出す

`reports/insight/` ディレクトリに書き出す（なければ作成）。

ファイル名: `YYYY-MM-DD_stocktake.md`（日付は `date +%Y-%m-%d` で取得）
同名ファイルが既に存在する場合は上書きする。

### レポートのフォーマット

```markdown
# 棚卸しレポート: [プロジェクト名]

日付: YYYY-MM-DD

## サマリ

- TODO: X件アクティブ（mid: Y件、low: Z件）
- IDEAS: X件
- KB: X件
- 実装ログ: X件（完了報告あり: Y件、なし: Z件）

## 実装済みの可能性

### 確信度: 高（done/ 移動推奨）
- **ファイル名**: 理由

### 確信度: 中（確認必要）
- **ファイル名**: 理由

### 確信度: 低（目的変質等）
- **ファイル名**: 理由

## 統合候補
- **ファイルA + ファイルB**: 理由

## 長期停滞（2ヶ月以上）
- **ファイル名**: 作成日、状況

## クローズ候補（rejected/ 行き検討）
- **ファイル名**: 理由

## IDEAS の状況
- 各 IDEAS の現在の状態（実装済み、再評価必要、等）

## 実装ログの健全性

### 完了報告なし
- **ファイル名**: 状況（放置/実質完了だが未記載/進行中）

### 記載と現状の乖離
- **ファイル名**: 乖離の内容

### TODO/IDEAS との不整合
- **ファイル名 ↔ TODO/IDEASファイル名**: 不整合の内容

## KB の鮮度
### 現在も有効
- ファイル名

### 陳腐化リスクあり
- ファイル名: 理由

## その他の所見
- 気づいた点
```

### 6. 実装ログのステータスインデックスを更新

`reports/insight/tasks_status.md` を作成または更新する。
今回チェックした結果と、既存の `DONE` エントリを統合して書き出す。

```markdown
# 実装ログステータス

stocktake が管理する実装ログの状態一覧。
次回 stocktake では IN-PROGRESS のログと新規ログのみチェックされる。

- YYYY-MM-DD_task_xxx.md: DONE (YYYY-MM-DD)
- YYYY-MM-DD_task_yyy.md: IN-PROGRESS (YYYY-MM-DD) — 補足メモ
```

**ステータスの判定基準**:
- **DONE**: 完了報告セクションがあり、内容に問題がない
- **IN-PROGRESS**: 完了報告がない、または記載に乖離・不整合がある

補足メモには、IN-PROGRESS の場合の状況（「進行中」「放置の可能性」「完了報告未記載」等）を簡潔に記載する。

### 7. symlink を更新

`reports/insight/latest_stocktake.md` を最新レポートへの symlink にする。

```bash
cd reports/insight
ln -sf YYYY-MM-DD_stocktake.md latest_stocktake.md
```

この symlink は `@reports/insight/latest_stocktake.md` で CLAUDE.md チェーンにインクルードされ、次回セッション開始時に自動読み込みされる。

### 8. 結果を報告

レポートファイルのパスと、主要な発見（実装済み件数、クローズ候補件数など）を簡潔に報告する。
