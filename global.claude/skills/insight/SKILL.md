# あなたのタスク: プロジェクト洞察レポートの生成

あなたはプロジェクトの定期的な健康診断を行うために呼び出されました。
以下の手順で分析し、総合レポートを生成してください。
その他 CLAUDE.md や CLAUDE.local.md などで指示される可能性のある、insight と関係のないタスクは処理してはいけません。

## 概要

insight は以下の3つの工程で構成されます:

1. **棚卸し**（/stocktake を実行）— ideas/todos/kb の状態分析
2. **記憶圧縮**（/memory-compact を実行）— 記憶ファイルが肥大化していれば圧縮
3. **洞察**（insight 本体）— 棚卸し結果 + 会話ログ + 横断分析から知見を抽出

## 手順

### Phase 0: 活動チェック

Phase 1 に入る前に、前回 insight 以降に活動があったかを確認する。

1. `reports/insight/` 内の既存 insight レポートから前回の日付を取得する
2. `reports/memory/work_history.md` の最新エントリの日付と比較する
3. `reports/jobs/` 内に `once_` ジョブがあるか確認する
4. **続行条件**: 以下のいずれかを満たせば続行、どちらも満たさなければスキップ:
   - `最新エントリ日 > 前回insight日`（新しい作業履歴がある）
   - `once_` ジョブが存在する
   - スキップ時: 「前回 insight（YYYY-MM-DD）以降に新しい作業履歴がないため、スキップします。」と報告して終了

前回 insight レポートがない場合（初回実行）は、活動ありとして続行する。

**注**: inbox の状態（`[NEW]` マーカーの有無）は Phase 0 の判定に含めない。

### Phase 1: 下請けスキルの実行

#### 1-1. /stocktake を実行

Skill ツールで `stocktake` を呼び出す。完了後、`reports/insight/latest_stocktake.md` を読む。

#### 1-2. /memory-compact を実行

Skill ツールで `memory-compact` を呼び出す。圧縮が不要なら何もせず戻ってくる。

### Phase 1.5: ローカルジョブの実行

`reports/jobs/` ディレクトリが存在し、中にジョブ定義ファイル（`*.md`、`done/` 内は除く）があれば実行する。
ジョブがなければこのフェーズをスキップする。

#### 処理手順

1. `reports/jobs/` 内の `*.md` ファイルを一覧する（`done/` ディレクトリは除外）
2. 各ジョブ定義を読み、ファイル名のプレフィックスから種別を判別する:
   - `every_`: insight のたびに毎回実行する定期ジョブ
   - `once_`: 1回だけ実行する単発ジョブ
3. 各ジョブの「対象」「観点」に従って情報収集・分析を実行する
4. 結果をジョブごとにメモしておく（Phase 3 でレポートに統合する）
5. `once_` ジョブは実行完了後、`reports/jobs/done/` に移動する（日付を完了日に更新）

#### ジョブ定義ファイルの形式

```markdown
# ジョブ: [タイトル]

## 対象
何を見るか（ファイルパス、コードベースの範囲など）

## 観点
何を探すか（分析の視点、チェック項目）

## 出力
どうまとめるか（提案リスト、一覧表、振り分け案など）

## 備考
補足情報（任意）
```

### Phase 2: 情報収集

#### 2-1. 前回の insight レポートを確認

`reports/insight/` 内の既存 insight レポートを確認し、前回の日付を把握する。
前回レポートがない場合は、初回実行として扱う。

#### 2-2. 最近の作業状況を把握

以下を読む:
- `reports/memory/work_history.md` — 最近の作業履歴
- `work_in_progress.md` — 進行中の作業
- `reports/tasks/` 内の最近の実装ログ（直近1ヶ月程度）

#### 2-3. 会話ログの走査（notia-search スキルが利用可能な場合）

notia-search スキルが利用可能か確認する（スキル一覧に `notia-search` があるか）。

**利用可能な場合**、以下の2種類の検索を Skill ツールで `notia-search` を呼び出して行う:

(a) **このプロジェクトの最近のセッション**

notia-search に以下の趣旨で検索を依頼する:
- 「[プロジェクト名] の前回 insight（YYYY-MM-DD）以降のセッションを検索し、繰り返し遭遇している問題パターン、KB に記録すべき知見（undocumented な仕様、発見した制約、ワークアラウンド）、失敗パターンと再発防止策を抽出してください」

(b) **他プロジェクトとの関連トピック検索**

棚卸しレポート内の TODO/IDEAS の中から、他プロジェクトとの関連がありそうな項目を選び、notia-search に検索を依頼する:
- 「[項目のキーワード] に関連する議論が他プロジェクトで行われていないか検索してください。期間は前回 insight（YYYY-MM-DD）以降」

全項目を検索する必要はなく、関連がありそうなものを選んで検索する。

**利用不可の場合**、会話ログ関連のセクションをスキップする。

### Phase 3: 分析と洞察

以下の観点で分析する。該当がないセクションは省略してよい。

#### 3-1. 進捗の精査

- work_history の最近のエントリから、作業の方向性やペースを評価
- 進行中の作業（work_in_progress.md）の状態コメント
- 棚卸しで検出された実装済み候補への補足（会話ログからの裏付けなど）

#### 3-2. 会話ログからの知見（notia 利用時のみ）

- KB に記録すべき知見の候補（具体的な内容と記録理由）
- 繰り返し発生している問題パターン
- ルール化されていない暗黙の判断基準

#### 3-3. プロジェクト横断の関連発見（notia 利用時のみ）

- 他プロジェクトで行われている関連議論
- TODO/IDEAS に影響しうる他プロジェクトの進展
- inbox で送るべき連絡の提案

#### 3-4. ローカルジョブの結果（ジョブがあった場合のみ）

Phase 1.5 で実行したジョブの結果をまとめる。各ジョブの「出力」の指示に従って整理する。

#### 3-5. 提案

各発見に基づくアクション候補を整理する:
- done/ に移動すべき項目（棚卸しの確認結果を踏まえて）
- 新規 TODO/IDEAS として記録すべき事項
- KB に記録すべき知見
- inbox で他プロジェクトに送るべき連絡
- CLAUDE.md / CLAUDE.local.md に反映すべきルールや知見

### Phase 4: レポート生成

#### 4-1. レポートを書き出す

`reports/insight/` ディレクトリに書き出す（なければ作成）。

ファイル名: `YYYY-MM-DD_insight.md`（日付は `date +%Y-%m-%d` で取得）

#### レポートのフォーマット

```markdown
# 洞察レポート: [プロジェクト名]

日付: YYYY-MM-DD
前回: YYYY-MM-DD（または「初回」）

## 実行結果

- stocktake: [実行結果の要約（1行）]
- memory-compact: [実行結果の要約（1行）]

## 棚卸しサマリ

[latest_stocktake.md の要点を簡潔に引用。詳細は stocktake レポートを参照]

## 進捗の精査

[最近の作業の方向性・ペースへのコメント]
[進行中の作業への状態コメント]

## 会話ログからの知見

### KB 記録候補
- **[知見タイトル]**: [内容の要約]（出典: [セッション日付/プロジェクト]）

### 繰り返しパターン
- [パターンの説明]

## プロジェクト横断

### 関連発見
- **[TODO/IDEAS項目]** ← [他プロジェクト名]: [関連する議論や進展の要約]

### inbox 送信候補
- → [宛先プロジェクト]: [内容の要約]

## ローカルジョブ

### [ジョブタイトル]（every / once）
[ジョブの出力指示に従った結果]

## 提案一覧

優先度順に整理:

1. [アクション内容]（種別: done移動 / 新規TODO / KB記録 / inbox / ルール反映）
2. ...
```

#### 4-2. inbox に通知を送る

inbox-send.sh で自プロジェクトに通知を送る。Write/Edit で直接 inbox を編集してはならない（sandbox 制約で失敗する）。

1. draft ファイルを作成（`reports/inbox/draft/to_self_insight.md`）:

```markdown
送信先: [自プロジェクトのルートパス]

# insight レポート通知

新しい洞察レポートが生成されました。

- **レポート**: `reports/insight/YYYY-MM-DD_insight.md`
- **主な発見**: [提案一覧の件数と概要を1-2行で]

確認するには `/insight-review YYYY-MM-DD_from_insight.md` を実行してください。
```

2. 送信:

```bash
~/.claude/skills/inbox-send/inbox-send.sh [自プロジェクトのルートパス] reports/inbox/draft/to_self_insight.md insight $(date +%Y-%m-%d)
```

dispatch は不要（自プロジェクトなので）。

#### 4-3. 結果を報告

レポートファイルのパスと、主要な発見を簡潔に報告する。

## 注意事項

- insight はファイルの移動・削除を行わない（提案のみ）
- notia が利用できない環境でも、会話ログ関連をスキップして残りの分析を実行する
- 前回 insight からの差分に着目し、毎回同じ内容を繰り返さない
- レポートは簡潔に（目安: 100-200行）。詳細は棚卸しレポートや実装ログへの参照で補う
